Keep 支持各种认证提供商和架构，以适应不同的部署策略和安全需求，从开发环境到生产设置。


### 认证提供商

- [**无认证**](/deployment/authentication/no-auth) - 快速设置，用于测试或内部使用场景。
- [**数据库**](/deployment/authentication/db-auth) - 简单的用户名/密码认证。适用于小团队或开发/预发布环境。用户和哈希密码存储在数据库上。
- [**Auth0**](/deployment/authentication/auth0-auth) - 使用 Auth0 实现可扩展的、基于 Auth0 的认证。
- [**Keycloak**](/deployment/authentication/keycloak-auth) - 使用 Keycloak 实现企业认证方法，如 SSO/SAML/OIDC、具有自定义角色的高级 RBAC、基于资源的权限以及与用户目录（LDAP）的集成。
- [**AzureAD**](/deployment/authentication/azuread-auth) - 使用 Azure AD 实现 SSO/SAML/OIDC 企业认证。
- [**Okta**](/deployment/authentication/okta-auth) - 使用 Okta 实现 SSO/OIDC 认证。
- [**OneLogin**](/deployment/authentication/onelogin-auth) - 使用 OneLogin 实现 SSO/OIDC 认证。

选择正确的认证策略取决于您的具体使用场景、安全要求和部署环境。您可以阅读更多关于每个认证提供商的信息。

### 如何配置
<Tip>
某些认证提供商需要额外的环境变量。这些将在特定的认证提供商页面上详细说明。
</Tip>
Keep 上的认证方案通过环境变量控制，包括后端（Keep API）和前端（Keep UI）。


|  身份提供商  | 环境变量 | 需要的额外变量 |
| ------------------------------------- | -------------------------------------------------------------- | ---------------------------- |
| **无认证** | `AUTH_TYPE=NOAUTH`| 无 |
| **数据库** | `AUTH_TYPE=DB`  | `KEEP_JWT_SECRET` |
| **Auth0**                           |  `AUTH_TYPE=AUTH0`  | `AUTH0_DOMAIN`, `AUTH0_CLIENT_ID`, `AUTH0_CLIENT_SECRET` |
| **Keycloak** | `AUTH_TYPE=KEYCLOAK` | `KEYCLOAK_URL`, `KEYCLOAK_REALM`, `KEYCLOAK_CLIENT_ID`, `KEYCLOAK_CLIENT_SECRET` |
| **Oauth2Proxy** | `AUTH_TYPE=OAUTH2PROXY` | `OAUTH2_PROXY_USER_HEADER`, `OAUTH2_PROXY_ROLE_HEADER`, `OAUTH2_PROXY_AUTO_CREATE_USER` |
| **AzureAD** | `AUTH_TYPE=AZUREAD` | 请参阅 [AzureAD 配置](/deployment/authentication/azuread-auth) |
| **Okta** | `AUTH_TYPE=OKTA` | `OKTA_DOMAIN`, `OKTA_CLIENT_ID`, `OKTA_CLIENT_SECRET` |
| **OneLogin** | `AUTH_TYPE=ONELOGIN` | 请参阅 [OneLogin 配置](/deployment/authentication/onelogin-auth) |

有关每种认证策略的更多详细信息，包括设置说明和影响，请参阅相应部分。
