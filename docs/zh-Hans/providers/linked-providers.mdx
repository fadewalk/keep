# 链接的 Provider

在 Keep 中，Provider 可以是"已连接的"或"链接的"。理解两者之间的区别对于正确的告警路由和管理非常重要。

</Frame>

## 已连接 vs 链接的 Provider

- **已连接的 Provider**：这些是已通过 UI 或 API 在 Keep 中明确配置的 Provider。它们具有完整的 Provider 配置和身份验证详细信息。

- **链接的 Provider**：这些是发送告警到 Keep 但未明确连接的 Provider。当 Keep 通过 webhooks 或推送机制接收来自它们的告警时，它们会自动出现。

## 链接的工作原理

当 Keep 从未连接的 Provider（如 Prometheus 推送告警）接收告警时，它会自动创建一个"链接的"Provider 条目。这允许您：

- 跟踪哪些系统正在发送告警
- 查看 Keep 上次接收告警的时间
- 应用特定于该 Provider 的去重规则

## 将告警附加到已连接的 Provider

如果您有一个已连接的 Provider，并希望将传入的告警与之关联而不是创建链接的 Provider，请将 `provider_id` 查询参数添加到 webhook URL。

例如，使用 Prometheus AlertManager：

```yaml
alertmanager:
  config:
    receivers:
      - name: "keep"
        webhook_configs:
          - url: "https://api.keephq.dev/alerts/event/prometheus?provider_id=your_provider_id"
```

或使用其他基于 webhook 的集成：

```bash
# Grafana webhook
https://api.keephq.dev/alerts/event/grafana?provider_id=grafana-prod

# Datadog webhook
https://api.keephq.dev/alerts/event/datadog?provider_id=datadog-main

# Generic webhook
https://api.keephq.dev/alerts/event/webhook?provider_id=custom-webhook
```

## 最佳实践

1. **对于生产系统**：建议尽可能使用已连接的 Provider，因为它们提供：

   - 更好的身份验证和安全性
   - 访问特定于 Provider 的功能
   - 更清晰的审计跟踪

2. **对于测试/开发**：链接的 Provider 可用于：

   - 快速原型设计
   - 测试告警流程
   - 临时集成

3. **将链接的转换为已连接的**：如果您定期从链接的 Provider 接收告警，请考虑：

   - 设置正确的 Provider 连接
   - 使用 `provider_id` 参数将告警附加到已连接的 Provider

## 限制

链接的 Provider：

- 不能用于拉取告警或数据
- 没有身份验证详细信息
- 不能用于特定于 Provider 的操作
- 可能具有有限的去重功能

要获得完整功能，请在链接的 Provider 成为永久告警基础设施的一部分时，考虑将其转换为已连接的 Provider。
